# 設計チェックリスト

## 要件定義

### 機能要件
- [ ] ユーザーストーリーが定義されている
- [ ] 機能一覧が優先度付きで整理されている
- [ ] 受け入れ基準が明確である
- [ ] 画面一覧が作成されている

### 非機能要件
- [ ] パフォーマンス目標が定義されている（画面表示X秒以内等）
- [ ] 対応OS/バージョンが明確
- [ ] オフライン対応の要否が決まっている
- [ ] セキュリティ要件が定義されている

---

## アーキテクチャ

### レイヤー設計
- [ ] Clean Architectureの層分離が定義されている
- [ ] 依存関係の方向が明確（外→内）
- [ ] 各層の責務が明確

### モジュール設計
- [ ] Feature単位でモジュール分割されている
- [ ] 共有コンポーネントが識別されている
- [ ] 依存関係が循環していない

---

## データモデル

### Entity
- [ ] 全Entityが定義されている
- [ ] 必須/任意フィールドが明確
- [ ] 型が適切に定義されている
- [ ] freezedによるイミュータブル設計

### ER図
- [ ] Entity間の関係が定義されている
- [ ] 主キー/外部キーが明確

---

## API設計

### エンドポイント
- [ ] RESTful設計に準拠
- [ ] 全エンドポイントが一覧化されている
- [ ] HTTPメソッドが適切

### リクエスト/レスポンス
- [ ] リクエストボディが定義されている
- [ ] レスポンスフォーマットが定義されている
- [ ] エラーレスポンスが定義されている

---

## 状態管理

### Provider設計
- [ ] Provider一覧が作成されている
- [ ] Provider種類（FutureProvider, NotifierProvider等）が選択されている
- [ ] 依存関係が整理されている

### 状態遷移
- [ ] 状態遷移図が作成されている
- [ ] 異常系の状態が考慮されている

---

## セキュリティ

- [ ] 認証方式が決定されている
- [ ] トークン管理方針が決まっている
- [ ] 機密データの保存方法が決まっている
- [ ] 通信暗号化（HTTPS）が前提

---

## テスト計画

- [ ] テスト種別（unit/widget/integration）が決まっている
- [ ] カバレッジ目標が設定されている
- [ ] テストケース一覧が作成されている
